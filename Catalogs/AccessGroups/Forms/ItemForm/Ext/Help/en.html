Access group properties.

An access group defines actions with the application data available for the group members. Usually, access groups correspond with different job duties (or occupations) of the application users.  A user may be a member of one or more access groups. Together such groups represent their personal access settings.

Use an access group to define and administer the common settings of the access rights of a certain group of users with similar functions, e.g. Accountants.

To manage access, you need administrator rights. The user responsible for the access group can add or delete members of their access group on the Members tab.

An access group is always connected with one of the existing access group profile that usually combine multiple roles. When added to a group, the user gets all the roles specified in the profile of this group, e.g. the predefined Administrators access group is associated with the Administrator profile having the Full rights role. This role grants unrestrained access to all the data and belongs to all the application administrators listed in the Administrators access group.

Adding access group

For a new access group, start with creating its profile (recommended to make it first), users list and access restrictions.

  • Select Profile. It determines the rights (with a roles list) and options of "domain" rights restrictions (access types), e.g. restrictions to Organizations, Warehouses, etc. The access types from the profile will be added to the access group automatically. 
  • Proceeding from the objectives and restrictions for the existing access types, enter the Name of the access group (For more information, see access group purposes).
    • It is recommended to name an access group as a relevant profile in plural (in the Profile field). The name should also identify the access settings of the access group, e.g. on the basis of the Sales managers profile, you can introduce such access groups as Sales managers at X Company and Sales managers at Y Company.
  • A group can belong to one of the access group folders.
    • In the Group (folder) field, select a folder from the list. 
  • Click to select the Member type - according to the access group purpose, you can appoint members as Arbitrary members (you can include users and external users) or predefined users.
    • To restrict your choice to a specific list, select the values: Regular users, External users ("Demo:Counterparties"), External users ("Demo: Individuals"), External users ("Demo: Partners").
    • An access group may include the above users and external user groups (it may be necessary, e.g. for questioning).
    • If you select Arbitrary members, you can select different user types (regular and external users) for your group.
  • On the Members tab, complete a list of access group members.
  • On the Access restrictions tab, configure the additional settings of the access rights.
  • On the Description tab, you can enter detailed information about the access group.

Members

  • According to the selected Member type, this access member selection tab will provide the following user lists.
    • To add multiple members, click Select, to add one member, click More - Add. Select necessary access group members from the list. 
    • To delete one or more access group members, select them and click Delete.
  • Fill in the Responsible user field (select them from the Users list). The user marked Responsible in an access group can add or delete its members.

Access restrictions

  • On the Access restrictions tab, specify additional access settings. The access group restrictions depend on the profile restriction but get more particular. Sometimes profile access restrictions settings do not require details. In such cases, the Access restrictions tab is unavailable. 
  • The access types stated on this tab depend on the profile selected in the Profile field. The profile also determines how the access type setting will function: either it is read-only mode or editing is allowed, e.g. if the access groups profile includes a role for reading documents, Goods receipt, the users can view the documents of stated organizations. If the profile contains a role for adding and editing, you can edit such documents.
  • Use the Access kind field to define the database scope allowed for the access group members. Access kind is a certain rule determining the access to the application data, e.g. the Organization access kind will allow operations only with the documents related to specific organizations or hide the documents of some organizations. 
  • To configure restrictions, define a list of values for each access kind.
  • Double-click to state your values (or perform the Edit command of the left-click context menu). Define the Access value for each Access kind. You can use one of the following values: 
    • All prohibited. You can state a list of allowed values for an access kind. This configuration method is good when you need to grant access only to a specific data domain.
    • All allowed. You can state a list of the values that will not be accessible for a certain access kind. This method is suitable when you need to close access to some predefined data domain, while all other existing and new values will be accessible, e.g. the Organizations access kind states a rule prohibiting operations with the documents of X Company. If a new organization, Y Company, is added to the application later, this rule will not cover it. Note that allowing settings prevail the forbidding ones. IN the above example if the user is allowed to add documents for X Company in some other group with the same profile, this user will be allowed to access all the documents of the organization. 
  • When necessary, each Access kind can have a values list or a values groups list that will state all the exceptions. Click Add in the Allowed (Forbidden) values list. Select exceptions from the proposed lists.
    • Sometimes, you can simplify the procedure by stating entire item groups in the allowed/forbidden values list instead of separate items, e.g. the Partners access kind can be restricted with the entire partner domain, instead of separate partners. To do it, specify the list of groups accessible for the partners and associate each partner with a certain group. Besides the administrator, any other user is permitted to edit the partner list.
  • In some access groups, you can set rights with the Users access kind (similarly, the External users for external users). This setting is valid only in the documents (and other application objects) which fields can relate to the users getting access to these documents ("Responsible", "Author", "Performer", etc.)
    • To permit all the access group members to access all the restricted data being processed by a certain user - add this user to the allowed values for the Users access kind.
    • For example, access to the Goods and services receipt documents depends on who is specified in the Responsible field. IN this case, the user can record the documents which Responsible user is that very user or is added to the Users access kind. The forbidding setting functions similarly.
