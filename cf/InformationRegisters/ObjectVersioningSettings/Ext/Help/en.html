Designed to configure object versioning.

You can version documents and catalogs to trace review history.

To maintain object revision history, click Support and maintenance in the Administration section and select the Object versioning check box (it is selected by default).

After that, you can click Storage settings in Administration - Support and maintenance and thus configure necessary settings.

The list includes:

  • Object - a project object list as a tree, the objects are grouped by type.
  • Versioning variant - the selected versioning variant.
  • Version storage period - the selected storage period for versions.

Selecting versioning variant

  • Select one or more objects and click Define versioning variant. IN the drop-down list, select the necessary versioning variant  for each document/catalog type (or find the relevant command in the More menu or in the context menu):
    • Do not version - do not maintain object version history.
    • Version when save - add a new record to the version history every time you edit (or change) a document/catalog.
    • Version when posting - save versions only when posting; allowed only for document allowed to be posted.
    • Version when starting - save versions only when a business process is started; allowed for all business processes.
    • Default - select the recommended settings (for catalogs - do not version, for documents - version when posting).
  • You can also set up versioning for a group of application objects, e.g. select all documents or catalogs.

Selecting version storage period

  • Click Define version storage period. From the drop-down list, select a storage period for each document/catalog type: Last week, Last month, Last three months, Last six months, Last year, and Dead storage.

Viewing quantity and volume of stored object versions report

  • Click the relevant link to view a Quantity and volume of stored object versions report.

Clearing outdated versions

  • The Outdated versions, total field shows the quantity and volume of outdated versions in the application.
  • Click Clear to delete outdated versions.
  • To Delete outdated versions automatically, select the relevant check box.
  • Click the relevant link to Set schedule. The configured schedule is shown at the bottom of the window.

Advantages of versioning

  • In the beginning of application implementation, information volumes are rather moderate and performers might make a lot of mistakes (e.g. enter unnecessary information or delete the necessary one). Then versioning can help to arrange lists and documents, correct errors and help performers to understand and prevent mistakes.
  • Versioning allows you to return to the previous document version or catalog item that was the most comprehensive unlike the last variant.
  • Later, when information volumes grow, you can consistently stop versioning some documents and save versions of only important elements, e.g. when posting documents. 
  • Versioning of a lot of objects may lead to excessive information volumes stored in the application, so the application response time may be seriously affected. Therefore, this option should be used only when necessary.

More about version when posting mode

  • If a document supports the Version when posting mode, the first version will be recorded in history only after posting the document. Later, the document versions will be recorded every time a posted document version is saved. IN this mode, you can not create versions of new or uncompleted documents, so application memory will be not be used. This option is recommended for all documents.

More about version when starting mode

  • If Version when starting is selected for a business process, its first version will be saved only after its start. This mode prevents from creating unnecessary business process versions while the business processes are being prepared for start.
